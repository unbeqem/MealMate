**File: `supabase/migrations/00002_enable_rls.sql`**

Enable RLS on all 4 tables immediately. Create SELECT/INSERT/UPDATE/DELETE policies for each table using the `(SELECT auth.uid())` pattern (the `SELECT` wrapper caches the function call for ~95% performance improvement per Supabase docs).

For EACH table, create 4 policies:
- SELECT: `USING ((SELECT auth.uid()) = user_id)`
- INSERT: `WITH CHECK ((SELECT auth.uid()) = user_id)`
- UPDATE: `USING ((SELECT auth.uid()) = user_id) WITH CHECK ((SELECT auth.uid()) = user_id)`
- DELETE: `USING ((SELECT auth.uid()) = user_id)`

Add user_id indexes on all tables for RLS query performance:
```sql
CREATE INDEX idx_ingredients_user_id ON ingredients(user_id);
CREATE INDEX idx_recipes_user_id ON recipes(user_id);
CREATE INDEX idx_meal_plan_slots_user_id ON meal_plan_slots(user_id);
CREATE INDEX idx_shopping_list_items_user_id ON shopping_list_items(user_id);
```

**Do NOT forget `WITH CHECK` on INSERT and UPDATE policies.** Missing `WITH CHECK` on INSERT lets any user insert rows attributed to other users. This is the second most common Supabase security bug.

**File: `supabase/migrations/00003_gdpr_deletion.sql`**

Create a PostgreSQL function `delete_user_data(target_user_id UUID)` that deletes all rows from all 4 tables for a given user_id. This is a scaffold for Phase 9's GDPR compliance work.

```sql
CREATE OR REPLACE FUNCTION delete_user_data(target_user_id UUID)
RETURNS void AS $$
BEGIN
  DELETE FROM shopping_list_items WHERE user_id = target_user_id;
  DELETE FROM meal_plan_slots WHERE user_id = target_user_id;
  DELETE FROM recipes WHERE user_id = target_user_id;
  DELETE FROM ingredients WHERE user_id = target_user_id;
END;
$$ LANGUAGE plpgsql SECURITY DEFINER;
```

Order matters: delete from dependent tables first (shopping_list_items, meal_plan_slots) before parent tables (recipes, ingredients).
