Fix two bugs in ingredient_search_provider.dart (Gap 1):

**Bug 1 — Local match inside debounce Timer:**
Currently the local `commonIngredients` check is inside the 300ms Timer callback. Users always wait 300ms even for "chicken" which has 5+ local matches.

**Fix:** Move the local check BEFORE the Timer. If >= 5 local matches, emit immediately and return (skip Timer entirely). Only start the debounce Timer for the API fallback path:

```dart
Future<void> search(String query) async {
  _debounce?.cancel();
  if (query.length < 2) {
    state = const AsyncData([]);
    return;
  }

  // LOCAL-FIRST: check curated list BEFORE debounce (instant)
  final localResults = commonIngredients
      .where((name) => name.toLowerCase().contains(query.toLowerCase()))
      .take(10)
      .toList();

  if (localResults.length >= 5) {
    // Fast path — enough local matches, emit immediately, no API call
    state = AsyncData(localResults);
    return;
  }

  // Emit local results immediately if any, then debounce for API
  if (localResults.isNotEmpty) {
    state = AsyncData(localResults);
  } else {
    state = const AsyncLoading();
  }

  // FALLBACK: OFf API for less common items (debounced)
  _debounce = Timer(const Duration(milliseconds: 300), () async {
    state = await AsyncValue.guard(() async {
      final apiResults = await ref
          .read(ingredientRepositoryProvider)
          .searchSuggestions(query);
      final combined = <String>{...localResults, ...apiResults};
      return combined.take(20).toList();
    });
  });
}
```

**Bug 2 — ref.onDispose placement:**
Currently `ref.onDispose(() => _debounce?.cancel())` is called inside `search()`, which means it registers a new dispose callback on every keystroke. Move it to `build()`:

```dart
@override
FutureOr<List<String>> build() {
  ref.onDispose(() => _debounce?.cancel());
  return [];
}
```

Remove the `ref.onDispose` line from inside `search()`.

Actually, a simpler approach: use the favorites provider data (which now contains Ingredient objects from Plan 03-04 fix) plus a new dedicated mechanism. The cleanest solution is:

1. Watch `ingredientFilterProvider` to get active filters
2. Watch `ingredientFavoritesProvider` to get favorite Ingredient objects (which have dietaryFlags)
3. For search results that are NOT in favorites, we cannot filter by dietary flags (OFf getSuggestions returns bare strings with no metadata). This is a known limitation documented in the research.

**Pragmatic approach for dietary filtering on search:**
- When dietary filters are active AND search results are showing, display an info banner: "Dietary filters apply to category results. Search results may not reflect all filters."
- Apply filtering where possible: if a suggestion name matches a cached Ingredient in the favorites list, use those flags to filter. Otherwise, show the result (don't hide it — hiding unverified items would make search feel broken).

**For dietary badges on tiles (Gap 4):**
Build a lookup map from the favorites provider data. For each search suggestion, if the name matches a favorited/cached Ingredient, pass its dietaryFlags to IngredientTile. Otherwise, pass empty flags (no badges — same as current, but now favorites-enriched items WILL show badges).

Implementation in the `data:` callback:
```dart
data: (suggestions) {
  if (suggestions.isEmpty) {
    return _buildEmptyOrBrowseState(context);
  }

  final activeFilters = ref.watch(ingredientFilterProvider);
  final favoritesList = favoritesAsync.value ?? [];
  // Build lookup: lowercase name -> Ingredient
  final ingredientLookup = <String, Ingredient>{};
  for (final fav in favoritesList) {
    ingredientLookup[fav.name.toLowerCase()] = fav;
  }

  // Build favorite name set for isFavorite display
  final favoriteNames = favoritesList
      .where((f) => f.isFavorite)
      .map((f) => f.name.toLowerCase())
      .toSet();

  // Apply dietary filter where we have data
  List<String> filtered = suggestions;
  if (activeFilters.isNotEmpty) {
    filtered = suggestions.where((name) {
      final cached = ingredientLookup[name.toLowerCase()];
      if (cached == null) return true; // Keep items without metadata (don't hide)
      return activeFilters.every((restriction) {
        final flag = _restrictionToFlag(restriction);
        return cached.dietaryFlags.contains(flag);
      });
    }).toList();
  }

  return ListView.builder(
    itemCount: filtered.length,
    itemBuilder: (_, i) {
      final name = filtered[i];
      final cached = ingredientLookup[name.toLowerCase()];
      final isSelected = selectedMap.containsKey(name);
      final isFav = favoriteNames.contains(name.toLowerCase());
      return IngredientTile(
        name: name,
        dietaryFlags: cached?.dietaryFlags ?? [],
        isSelected: isSelected,
        isFavorite: isFav,
        onFavoriteTap: () => ref
            .read(ingredientFavoritesProvider.notifier)
            .toggleFavorite(name, name: name),
        onSelectTap: () => ref
            .read(selectedTodayProvider.notifier)
            .toggle(name, name: name),
      );
    },
  );
},
```

Add the `_restrictionToFlag` helper method to the screen class (same as in ingredient_category_screen.dart):

String _restrictionToFlag(DietaryRestriction restriction) {
  switch (restriction) {
    case DietaryRestriction.vegetarian: return 'vegetarian';
    case DietaryRestriction.vegan: return 'vegan';
    case DietaryRestriction.glutenFree: return 'gluten-free';
    case DietaryRestriction.dairyFree: return 'dairy-free';
  }
}
