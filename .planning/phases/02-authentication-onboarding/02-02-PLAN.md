**1. Create Supabase profiles migration (`supabase/migrations/create_profiles_table.sql`):**
```sql
create table public.profiles (
  id uuid references auth.users on delete cascade primary key,
  household_size integer,
  dietary_preferences text[],
  onboarding_completed boolean default false,
  updated_at timestamptz default now()
);

alter table public.profiles enable row level security;

create policy "Users can manage their own profile"
  on public.profiles
  for all
  using (auth.uid() = id)
  with check (auth.uid() = id);

create index profiles_id_idx on public.profiles(id);

-- Auto-create empty profile row on signup (prevents race condition)
create or replace function public.handle_new_user()
returns trigger
language plpgsql
security definer set search_path = ''
as $$
begin
  insert into public.profiles (id)
  values (new.id);
  return new;
end;
$$;

create trigger on_auth_user_created
  after insert on auth.users
  for each row execute procedure public.handle_new_user();
```

This file is a reference for the user to run manually in the Supabase SQL editor. The `on_auth_user_created` trigger is CRITICAL — without it, a race condition exists where newly signed-up users have no profile row.

**IMPORTANT:** Use `text[]` array column for `dietary_preferences`, NOT a comma-separated `text` column. The Supabase Dart client handles `List<String>` to `text[]` serialization automatically.

**Update `lib/app/router.dart`** (created in Phase 1) to implement three-state routing:

**1. Create `RouterRefreshNotifier`:**
A `ChangeNotifier` that bridges Riverpod providers to go_router's `refreshListenable`:
```dart
class RouterRefreshNotifier extends ChangeNotifier {
  RouterRefreshNotifier(Ref ref) {
    ref.listen(authStateChangesProvider, (_, __) => notifyListeners());
    ref.listen(onboardingCompletedProvider, (_, __) => notifyListeners());
  }
}
```

**2. Update the `routerProvider`:**
The router provider (already created in Phase 1) must be updated to:
- Watch `authStateChangesProvider` for auth state
- Watch `onboardingCompletedProvider` for onboarding completion
- Use `RouterRefreshNotifier` as the `refreshListenable`

**3. Implement the `redirect` callback:**
Three-state logic (see RESEARCH Pattern 3):
```
if (authState is loading or error) → return null (don't redirect)
if (NOT authenticated):
  - if on /login, /signup, or /forgot-password → return null (stay)
  - else → redirect to /login
if (authenticated AND NOT onboarded):
  - if on /onboarding → return null (stay)
  - else → redirect to /onboarding
if (authenticated AND onboarded):
  - if on /login, /signup, /forgot-password, or /onboarding → redirect to /home
  - else → return null (stay)
```

**4. Register all auth and onboarding routes:**
Add these routes to the `GoRouter.routes` list:
- `/login` → `LoginScreen`
- `/signup` → `SignupScreen`
- `/forgot-password` → `ForgotPasswordScreen`
- `/reset-password` → `ResetPasswordScreen`
- `/onboarding` → `OnboardingShell`
- `/home` → home screen placeholder (can be a `Scaffold` with "Home" text if not yet built)

**5. Handle `passwordRecovery` AuthChangeEvent:**
In the redirect callback, check if the current auth event is `passwordRecovery`. If so, redirect to `/reset-password`. This handles the deep link flow: user taps reset email link → app opens → `passwordRecovery` event fires → router redirects to reset password screen.

**6. Sync onboarding flag on app startup:**
Create an `onboardingCompletedProvider` that:
- On first read: checks `SharedPreferences` for `onboarding_completed` boolean
- On `initialSession` event (app startup with existing session): fetches the profile row from Supabase (`supabase.from('profiles').select('onboarding_completed').eq('id', user.id).single()`) and updates the local `SharedPreferences` flag
- This handles the reinstall scenario where the local flag is gone but Supabase has the data

**ANTI-PATTERNS TO AVOID:**
- Do NOT check onboarding state only at app launch — the `refreshListenable` must be wired to BOTH auth and onboarding providers so the redirect re-evaluates on every state change
- Do NOT navigate to home directly in the `signedIn` handler — navigation is exclusively via go_router redirect
- Do NOT use `user_metadata` in any logic — only `auth.uid()` is safe
- Do NOT forget to handle the `initialSession` event for existing sessions — the redirect must work on cold start with an existing session, not just on fresh login
