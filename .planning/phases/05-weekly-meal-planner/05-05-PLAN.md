1. Create `ingredient_reuse_provider.dart` with two @riverpod providers:

   a. `weekIngredientNames(DateTime weekStart)` — returns `Set<String>`:
      - Watch `mealPlanNotifierProvider(weekStart)`.
      - For each filled slot (recipeId != null), load the CachedRecipe from the database by parsing the recipeId string to int.
      - Parse `jsonData` (JSON string) from CachedRecipe -> decode -> extract `extendedIngredients` array -> collect all `name` fields, lowercased.
      - Return the union Set of all ingredient names across all filled slots.
      - Handle: if a CachedRecipe is not found for a recipeId (e.g., cache expired), skip it silently.
      - This provider needs AppDatabase access: use `ref.watch(appDatabaseProvider)` to query cachedRecipes directly.
      - Implementation: since this needs async DB reads, make it a `@riverpod FutureOr<Set<String>>` provider (not sync). It watches the meal plan stream and recomputes when slots change.

   b. `ingredientOverlapCount({required DateTime weekStart, required List<String> candidateIngredientNames})` — returns `int`:
      - Watch `weekIngredientNamesProvider(weekStart)`.
      - Compute `weekNames.intersection(candidateNames.map((n) => n.toLowerCase()).toSet()).length`.
      - Return the count. If weekIngredientNames is still loading, return 0.

2. Hand-craft `ingredient_reuse_provider.g.dart` following the Riverpod 3.x codegen pattern for both providers. The first is a family FutureProvider with DateTime param. The second is a family provider with named params (weekStart + candidateIngredientNames).
