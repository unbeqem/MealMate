```dart
/// Returns the set of all ingredient names (lowercase) used across all
/// filled slots in the current week's plan.
@riverpod
Set<String> weekIngredientNames(Ref ref, DateTime weekStart) {
  final slotsAsync = ref.watch(mealPlanNotifierProvider(weekStart));
  return slotsAsync.whenData((slots) {
    final names = <String>{};
    for (final slot in slots) {
      if (slot.recipeId != null) {
        // Get ingredient names from the cached_recipes join data
        // The exact access pattern depends on Phase 4's cache structure:
        // If cached_recipes stores extendedIngredients as JSON, parse it
        // If summary-only (no ingredients), skip this slot
        final ingredients = slot.ingredientNames; // List<String>? from join
        if (ingredients != null) {
          names.addAll(ingredients.map((n) => n.toLowerCase()));
        }
      }
    }
    return names;
  }).valueOrNull ?? {};
}

/// For a candidate recipe, compute how many of its ingredients overlap with
/// the current week's ingredient set.
@riverpod
int ingredientOverlapCount(
  Ref ref, {
  required DateTime weekStart,
  required List<String> candidateIngredientNames,
}) {
  final weekIngredients = ref.watch(weekIngredientNamesProvider(weekStart));
  final candidateSet = candidateIngredientNames.map((n) => n.toLowerCase()).toSet();
  return weekIngredients.intersection(candidateSet).length;
}
```

**Handling summary-only cached recipes (research Pitfall 6):**
Phase 4 caches recipe browse results as summary-only entries (no `extendedIngredients`). The `weekIngredientNamesProvider` should check each slot's cached recipe to see if it has ingredient data. If not, the ingredient names for that slot are simply empty (the slot is skipped). No lazy-load trigger from this provider — lazy-loading full recipe data is Phase 4's responsibility when the user opens recipe detail. The overlap computation uses whatever ingredient data is already available locally.

For the `MealSlot` domain model to carry ingredient names, either:
- (a) Extend the `MealSlot` freezed class with a `List<String>? ingredientNames` field populated from the cached recipe's JSON data during the DAO join query, OR
- (b) Create a separate provider that reads ingredient names from cached_recipes for each recipeId in the week

Option (a) is cleaner. Update the `meal_plan_dao.dart` join query to also extract ingredient names from the cached recipe's JSON data column (if it has one). If the cached recipe model stores ingredients as a JSON string, parse it in `MealSlot.fromJoinRow()`.

```dart
class IngredientOverlapBadge extends StatelessWidget {

}
```
